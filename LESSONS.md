## データ構造の変更は必ず全箇所同時に

**何が起きたか:** Google Sheets の HEADERS（列の並び順）を `sheets-rest.ts` だけ変更し、`sheets.ts` を直し忘れた。結果、承認ステータスが隣の列に書き込まれ、Slack 承認がまったく反映されなくなった。

**教訓:** データの形式（列順、フィールド名など）を定義している箇所が複数ファイルにある場合、**必ず全箇所を同時に更新すること**。「1箇所だけ直してもう1箇所を忘れる」は最も起きやすい事故。

**対象ファイル:** `src/lib/sheets.ts` と `src/lib/sheets-rest.ts` の HEADERS 定義

## OAuth トークンは失効する前提で設計する

**何が起きたか:** Google API の OAuth Refresh Token が突然 `invalid_grant` エラーで失効。Sheets への全アクセスが停止した。

**教訓:** OAuth（ユーザーの代わりにログインする仕組み）は定期的に失効するリスクがある。自動化システムでは、ユーザー操作なしで認証が継続する**サービスアカウント方式**を使うべき。

**現在の対応:** サービスアカウント方式に移行済み。OAuth は使わない。

## 「今日」と「明日」の日付ズレに注意

**何が起きたか:** 承認時に `scheduled_date = today`（今日の日付）を設定していたが、投稿する cron（定時バッチ）は翌朝に動く。結果、承認したアイテムが永久に投稿されなかった。

**教訓:** 日付を使ったスケジューリングでは、「いつ書き込むか」と「いつ読み取るか」のタイミングを必ず確認する。承認 → 翌日投稿のフローなら `scheduled_date = tomorrowJst`。

## `.find()` は1件しか返さない

**何が起きたか:** daily-publisher で `.find()`（配列から条件に合う最初の1件だけを取り出す関数）を使っていたため、同日に同タイプ（blog x 2件など）がスケジュールされると2件目が無視された。

**教訓:** 「該当する全件を処理したい」場面では `.filter()`（条件に合うもの全部を取り出す関数）を使う。`.find()` は「1件だけ取りたい」ときだけ。

## Gemini API は 503 が頻発する前提で書く

**何が起きたか:** Gemini 2.5 Flash が高負荷時に 503 エラー（サーバーが混んでいる）を返し、ブログ生成が失敗した。

**教訓:** Gemini API を呼ぶコードには必ず**指数バックオフ retry**（失敗するたびに待ち時間を倍にしながら再試行する仕組み）を入れる。現在は最大10回リトライする設定。

**パターン:**
```typescript
async function withRetry<T>(fn: () => Promise<T>, maxRetries = 10, baseDelay = 15000): Promise<T> {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try { return await fn(); }
    catch (err: any) {
      if (attempt === maxRetries || err?.status !== 503) throw err;
      const delay = baseDelay * Math.pow(2, attempt - 1);
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error("Unreachable");
}
```

## Make.com の変数名を統一しないと環境構築で事故る

**何が起きたか:** `MAKE_IG_PUBLISH_WEBHOOK_URL` と `MAKE_PUBLISH_WEBHOOK_URL` が同じ用途なのに名前が違い、新しい環境で片方だけ設定して IG 投稿がスキップされた。

**教訓:** 同じ目的の環境変数は1つの名前に統一する。やむを得ず複数ある場合は fallback チェーン（片方がなければもう片方を試す仕組み）を必ず入れる。
